# Phase 999.8: 100% Unit Test Coverage - Research

**Researched:** 2026-04-02
**Domain:** Vitest + Vue 3 component/store/utility testing in a Bun monorepo
**Confidence:** HIGH

## Summary

This phase brings testing into scope for the nick-site monorepo. The codebase currently has zero tests and zero test infrastructure. The testable surface is modest: 36 source files in `apps/shell/src` and 8 in `packages/` (44 total). The most valuable test targets are 3 Pinia stores, 2 composables, 1 utility function, 9 theme definitions, and 13 Vue components (7 in shell, 6 in packages/ui).

Vitest 4.x is the standard testing framework for Vite + Vue 3 projects, created by the Vue/Vite team. It reuses the Vite config (plugins, aliases, transforms) so `.vue` files work without additional setup. For DOM simulation, happy-dom is the recommended default (2-4x faster than jsdom). Vue Test Utils v2.4 provides `mount`/`shallowMount` for component testing. `@pinia/testing` provides `createTestingPinia` for store isolation. Coverage uses the built-in V8 provider (`@vitest/coverage-v8`) with threshold enforcement.

**Primary recommendation:** Install Vitest 4, happy-dom, @vue/test-utils, @pinia/testing, and @vitest/coverage-v8 as root-level dev dependencies. Use the `projects` config (not deprecated `workspace`) to run tests across both `apps/shell` and `packages/ui`. Set coverage thresholds to 100% and add a `test` script to the root package.json. Add a CI workflow step that runs `bun run test -- --coverage` before deploy.

### Vitest Configuration

Use root-level `vitest.config.ts` with `projects` (not the deprecated `workspace` from pre-3.2). Each project gets its own config inheriting from a shared base.

**Root `vitest.config.ts`:**

```typescript
import { defineConfig } from 'vitest/config'

export default defineConfig({
  test: {
    projects: ['apps/shell', 'packages/ui'],
  },
})
```

**`apps/shell/vitest.config.ts`:**

```typescript
import { defineConfig, mergeConfig } from 'vitest/config'
import viteConfig from './vite.config'

export default mergeConfig(
  viteConfig,
  defineConfig({
    test: {
      environment: 'happy-dom',
      globals: true,
      include: ['src/**/*.test.ts'],
      coverage: {
        provider: 'v8',
        include: ['src/**/*.{ts,vue}'],
        exclude: ['src/main.ts', 'src/**/*.d.ts', 'src/data/**'],
        thresholds: {
          lines: 100,
          functions: 100,
          branches: 100,
          statements: 100,
        },
      },
    },
  }),
)
```

### Pattern 1: Store Testing (Pure Logic -- No DOM)

**What:** Test Pinia stores directly without mounting components.
**When to use:** All 3 stores (skills, theme, app) -- they contain the core business logic.
**Example:**

```typescript
import { describe, it, expect, beforeEach, vi } from 'vitest'
import { setActivePinia, createPinia } from 'pinia'
import { useSkillsStore } from '@/stores/skills'

describe('useSkillsStore', () => {
  beforeEach(() => {
    setActivePinia(createPinia())
  })

  it('starts with "All" as the only active category', () => {
    const store = useSkillsStore()
    expect(store.activeCategories).toEqual(new Set(['All']))
  })

  it('toggleCategory: selecting individual deselects All', () => {
    const store = useSkillsStore()
    store.toggleCategory('Frontend')
    expect(store.activeCategories.has('All')).toBe(false)
    expect(store.activeCategories.has('Frontend')).toBe(true)
  })

  it('toggleCategory: deselecting last individual falls back to All', () => {
    const store = useSkillsStore()
    store.toggleCategory('Frontend')
    store.toggleCategory('Frontend')
    expect(store.activeCategories).toEqual(new Set(['All']))
  })
})
```

### Pattern 2: Component Testing (With Store Injection)

**What:** Mount Vue components with createTestingPinia for isolated testing.
**When to use:** All component tests that depend on Pinia stores.
**Example:**

```typescript
import { describe, it, expect } from 'vitest'
import { mount } from '@vue/test-utils'
import { createTestingPinia } from '@pinia/testing'
import SkillDiamond from '@/components/skills/SkillDiamond.vue'

describe('SkillDiamond', () => {
  const skill = {
    name: 'vue',
    displayName: 'Vue.js',
    iconPath: '/icons/vue.svg',
    category: 'Frontend',
    years: 3,
  }

  it('renders with full opacity when visible', () => {
    const wrapper = mount(SkillDiamond, {
      props: { skill, mode: 'uniform', diamondSize: 80 },
      global: {
        plugins: [createTestingPinia()],
      },
    })
    // isSkillVisible returns true by default (All active)
    expect(wrapper.find('[aria-label="Vue.js"]').classes()).toContain(
      'opacity-100',
    )
  })

  it('emits hover event on mouseenter', async () => {
    const wrapper = mount(SkillDiamond, {
      props: { skill, mode: 'uniform', diamondSize: 80 },
      global: {
        plugins: [createTestingPinia()],
      },
    })
    await wrapper.find('[aria-label="Vue.js"]').trigger('mouseenter')
    expect(wrapper.emitted('hover')).toHaveLength(1)
  })
})
```

### Pattern 3: Utility/Pure Function Testing

**What:** Direct import and test -- no DOM, no Vue context needed.
**When to use:** `debounce`, `features`, `themes/index`, `federation/remotes`.
**Example:**

```typescript
import { describe, it, expect, vi } from 'vitest'
import { debounce } from '@/utils/debounce'

describe('debounce', () => {
  it('delays execution by the specified ms', () => {
    vi.useFakeTimers()
    const fn = vi.fn()
    const debounced = debounce(fn, 200)
    debounced()
    expect(fn).not.toHaveBeenCalled()
    vi.advanceTimersByTime(200)
    expect(fn).toHaveBeenCalledOnce()
    vi.useRealTimers()
  })
})
```

### Pattern 4: Composable Testing

**What:** Test composables that use Vue lifecycle hooks within a component context.
**When to use:** `useIntersectionObserver`, `useTheme`.
**Example:**

```typescript
import { describe, it, expect, vi, beforeEach } from 'vitest'
import { ref, nextTick } from 'vue'
import { mount } from '@vue/test-utils'
import { createTestingPinia } from '@pinia/testing'
import { useTheme } from '@/composables/useTheme'

// useTheme must be called inside a component (it uses watch + store)
describe('useTheme', () => {
  it('applies CSS variables to document.documentElement on init', () => {
    const TestComponent = {
      setup() {
        useTheme()
        return {}
      },
      template: '<div />',
    }

    mount(TestComponent, {
      global: { plugins: [createTestingPinia()] },
    })

    const root = document.documentElement.style
    // Should have applied --color-surface from the default theme
    expect(root.getPropertyValue('--color-surface')).toBeTruthy()
  })
})
```

### Anti-Patterns to Avoid

- **Testing implementation details:** Don't assert on internal refs or private state. Test behavior (what the user sees/does).
- **Snapshot-heavy tests:** Snapshots break on every markup change. Prefer targeted assertions on classes, text, emitted events.
- **Not cleaning up stores:** Always use `setActivePinia(createPinia())` in `beforeEach` or use `createTestingPinia` per test.
- **Importing `@/stores/theme` in packages/ui tests:** packages/ui components import from `@/stores/theme` which resolves to `apps/shell/src/stores/theme` -- the alias must be configured in the packages/ui vitest config.

## Don't Hand-Roll

| Problem                         | Don't Build                  | Use Instead                                     | Why                                                              |
| ------------------------------- | ---------------------------- | ----------------------------------------------- | ---------------------------------------------------------------- |
| Pinia store mocking             | Manual mock factories        | `createTestingPinia`                            | Handles action stubbing, initial state, spy wiring               |
| DOM simulation                  | Custom window/document stubs | happy-dom                                       | Full DOM API, event dispatch, CSS parsing                        |
| Timer mocking                   | setTimeout wrappers          | `vi.useFakeTimers()`                            | Built-in Vitest; controls Date, setTimeout, setInterval          |
| Coverage reporting              | Custom file instrumenters    | `@vitest/coverage-v8`                           | AST-based V8 remapping, threshold enforcement, JSON/HTML reports |
| Component mounting with plugins | Manual createApp + mount     | `@vue/test-utils mount()` with `global.plugins` | Handles plugin injection, async setup, event simulation          |

## Common Pitfalls

### Pitfall 1: packages/ui imports from `@/stores/*`

**What goes wrong:** packages/ui components (ThemeDropdown, MobileMenu) import `@/stores/theme` and `@/themes`. These are shell app paths, not package-local paths.
**Why it happens:** The UI package is designed to be consumed by the shell app which provides these aliases via vite.config.ts.
**How to avoid:** The packages/ui vitest.config.ts MUST replicate the same `@` alias pointing to `../../apps/shell/src`. Otherwise all component tests in packages/ui will fail to resolve imports.
**Warning signs:** "Cannot find module @/stores/theme" errors during test runs.

### Pitfall 2: Vitest `globals: true` and @pinia/testing spy detection

**What goes wrong:** `createTestingPinia` auto-detects the test framework spy function. Without `globals: true`, it cannot find `vi.fn` and action stubbing silently fails.
**Why it happens:** @pinia/testing checks for a global `vi` or `jest` object to auto-configure createSpy.
**How to avoid:** Either set `globals: true` in vitest config (recommended) or pass `createSpy: vi.fn` explicitly to createTestingPinia.
**Warning signs:** Store actions executing real code during tests instead of being stubbed.

### Pitfall 3: localStorage in theme store

**What goes wrong:** `useThemeStore` reads `localStorage` on creation. Tests run in happy-dom which provides a localStorage stub, but residual state leaks between tests.
**Why it happens:** happy-dom's localStorage persists within a test file unless explicitly cleared.
**How to avoid:** Call `localStorage.clear()` in `beforeEach` for any test file that touches theme state.
**Warning signs:** Tests pass individually but fail when run together.

### Pitfall 4: Coverage excluding the right files

**What goes wrong:** Coverage reports include `main.ts`, JSON data files, or type declaration files, making 100% unreachable.
**Why it happens:** V8 instruments everything that Vite processes unless explicitly excluded.
**How to avoid:** Set `coverage.exclude` to include `main.ts`, `*.d.ts`, `data/**`, and `env.d.ts`.
**Warning signs:** Coverage report shows uncoverable lines in app bootstrap or type files.

### Pitfall 5: Vitest projects config inheritance

**What goes wrong:** Project-level vitest configs don't inherit from the root config.
**Why it happens:** Since Vitest 3.2, projects are isolated -- root config only defines which projects exist.
**How to avoid:** Use `mergeConfig` to combine the existing vite.config.ts with vitest test options. Do NOT try to extend the root vitest.config.ts from a project config.
**Warning signs:** Aliases not resolving, plugins not loaded, "Cannot find module" errors.

### Pitfall 6: IntersectionObserver not available in happy-dom

**What goes wrong:** `useIntersectionObserver` composable uses `IntersectionObserver` which is not implemented in happy-dom.
**Why it happens:** happy-dom does not implement all Web APIs. IntersectionObserver is a common gap.
**How to avoid:** Create a global mock in a setup file: `vi.stubGlobal('IntersectionObserver', MockIntersectionObserver)`.
**Warning signs:** "IntersectionObserver is not defined" runtime errors in tests.

  unobserve: vi.fn(),
  disconnect: vi.fn(),

## Open Questions

1. **100% coverage on Vue template branches**
   - What we know: V8 coverage instruments compiled template output. Conditional rendering (`v-if`, `:class` ternaries) creates branches.
   - What's unclear: Some template edge cases (e.g., slot fallbacks, dynamic component resolution) may produce uncoverable branches in compiled output.
   - Recommendation: Start with 100% threshold. If specific files have truly uncoverable branches, use per-file threshold overrides rather than lowering the global threshold.

2. **packages/ui cross-dependency on shell app**
   - What we know: UI components import from `@/stores/*` and `@/themes/*` which live in apps/shell. This is an architectural coupling.
   - What's unclear: Whether this coupling will cause issues when more remotes are added (Phase 8).
   - Recommendation: For testing, replicate the alias. Note this as a future refactor candidate but do not address in this phase.
